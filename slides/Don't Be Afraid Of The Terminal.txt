# The Problem
### Computers think like this 👇
```
; **Register usage**
; R1 -- input data pointer
; R2 -- most recent value
;
x3000 **LD R1, x006**        ; load data pointer
x3001 **LDR R2, R1, #0**     ; load data value
x3002 **BRz x005**           ; branch to end if zero
;
; repeating statements go here
;
x3003 **ADD R1, R1, #1**     ; increment data pointer
x3004 **BRnzp x001**         ; branch back to top
x3005 **HALT**
;
; data section
;
x3006 **x4000** ; address of data
```

Historically, computers wanted extremely literal, low-level instructions. That made software creation feel inaccessible to most people.

---
### Humans think like this 👉

![](media/ChatGPT Image Mar 29, 2026, 10_57_30 AM.png)

Humans think in goals, stories, and rough intent. We start with outcomes, not memory addresses.

---
![](media/ChatGPT Image Mar 29, 2026, 10_54_45 AM.png)

This is the gap we used to bridge manually. A lot of programming was translating messy human intent into rigid machine steps.

---
### ==Development of Programming Languages==

Programming languages kept moving closer to human language and further from raw machine logic. Each step made computers more usable.

---
![](media/ChatGPT Image Mar 29, 2026, 10_56_04 AM.png)

Every layer of abstraction removed some pain. We did not make computers smarter; we made the interface kinder.

---

![](media/ChatGPT Image Mar 29, 2026, 11_00_25 AM.png)

Coding agents are the next step in that same story. Natural language is becoming a workable interface for building software.

# Setup
### We will start by building the structure of our application.


	```
	# App Setup

	I want to build a[n]

	[desktop app that runs on Mac and/or Windows.]
	[a mobile app that runs on iOS or Android]
	[a web app]
	[a Chrome browser extension]

	Here is what it needs to do:

	[YOUR BRIEF HERE]

	Based on my requirements, choose the most appropriate framework and
	approach for a project of this scale. Explain your choice in plain
	English before you start.

	Then:
	1. Scaffold the full project structure
	2. Install all dependencies
	3. Get it running so I can see the app window open on my machine
	4. Run /init to generate a [CLAUDE.md/AGENTS.md]
	5. Do not actually build the feature or project; we will do that in a later step
	6. Add the following block to the TOP of that [CLAUDE.md/AGENTS.md]

	---
	# About Me
	I have no coding or technical background. This may be the first time
	I've ever used a terminal or coding tool. Please treat me accordingly.

	# How to Communicate With Me
	- Explain what a command does BEFORE showing it to me
	- Translate error messages into plain English - don't assume I can read them
	- Give me one clear path forward, not options and tradeoffs
	- Use numbered steps when asking me to do something
	- Check in after each significant step before moving on

	Before any significant action, tell me: what we're doing,
	what will happen, and whether it can be undone.

	Tone: encouraging, jargon-free, treat all questions as normal questions.

	## My Goal
	I want to feel in control of what we're building - prioritise my
	confidence and comprehension over speed or technical elegance.
	---
	When everything is running, tell me: what I should see, where to find
	it, and what we're building next.
	```

This is the starter prompt I would give a beginner. It asks the agent to choose the stack, set things up, and communicate clearly.

# Build
### Now we build

	```
	## Mode: Build | Debug | Review | Explore

	## Goal
	[One clear sentence: what should exist or be true when you're done?]

	## Context
	- Relevant files: [list them or say "see src/auth/"]
	- Constraints: [e.g. "must not touch the DB layer", "iOS compatible"]
	- Related issue/ticket: [link or description]

	## What I've Already Tried (Debug mode only)
	[Error message, logs, what you ruled out]

	## Acceptance Criteria
	- [ ] [Specific, testable outcome]
	- [ ] [Another outcome]

	## Notes
	[Any edge cases, preferences, or things to watch out for]
	```

Once the project exists, prompts should become more structured. Clear goals, constraints, and acceptance criteria lead to much better output.

---
# Debug
### Something *will* break; no software is perfect.

	```
	## Mode:  Debug

	## Expected Behaviour
	A few short sentences about what you expected the software to do. Include what you were doing when the problem happened.

	## Current Behaviour
	A few short sentences about what is happening instead.

	## Context
	- Relevant files/paths: [list them or say "see src/auth/"]
	- Any error messages that you see when the problem occurs
	- Any log files that exist and can be checked
	- Related issue/ticket: [link or description]

	## What I've Already Tried
	A list of everything you've tried

	## Acceptance Criteria
	- [ ] [Specific, testable outcome]
	- [ ] [Another outcome]

	## Notes
	[Any edge cases, preferences, or things to watch out for]
	```

Debugging works best when you describe the gap between expected behaviour and actual behaviour. That gives the agent something concrete to investigate.
